## 1. デモ構造「I do → You do」（R8-01）

各タスクは **講師が玄米茶で5分デモ → 参加者が自分の題材で実行** の順で進める。
デモの目的は「完璧な最終形」を見せることではなく、**「AI にどう頼むか」** を見せること。うまくいかない場面もそのまま見せてよい（失敗も教材）。

**事前生成が要るもの**: [1] Claude Design は生成に数分かかる。**講師は玄米茶の3案を事前に生成**しておき、当日は「3案の提示 → 選択 → zip 書き出し」の流れを見せる（当日の待ち時間を作らない）。

| タスク | 5分デモで見せること |
|---|---|
| [1] デザイン | 事前生成した玄米茶の3案を提示 → 1案選択 → zip 書き出し。「機能や売り文句でなくサービスの中身を伝える」プロンプトの書き方 |
| [2] フロント | リポジトリのルートで Claude Code 起動 → zip を渡して「frontend/ に SPA を作って」→ `npm run dev` で表示 |
| [3] 公開 | `wrangler login` → `wrangler deploy` → 発行された URL をブラウザで開く |
| [4] バック | 「frontend/ を見て対応する API を backend/ に作って」の一括依頼 → D1/KV 作成 → ローカルで動作確認 |
| [5] テスト | 「テスト観点も出してからテストを書いて」→ `vitest` 実行 → グリーンを見せる |
| [6] 再公開 | `npm run deploy`（build:front → deploy）→ 公開 URL で確認 |

各デモの最後に「今の頼み方のコツ」を一言添えると、参加者の頼み方が上達する（発表会「一番効いた工夫」への伏線）。

## 3. リカバリー・巻きルール（R8-07）

AI 駆動は出力の個人差が大きく、詰まる人は必ず出る。**最低ライン**を死守し、間に合わない機能は割り切る。

**最低ライン（全員ここだけは到達）**
- Day 1: LP がローカル（`npm run dev`）で表示できる。
- Day 2: `workers.dev` で公開でき、API が最低1本つながっている。

**遅延時の割り切り**
- [1] が長引く → 玄米茶の zip をそのまま使って [2] へ進ませる（デザインは後で差し替え可）。
- [4] が時間内に終わらない → 問い合わせ or マイページのどちらか1機能に絞る。レート制限・KV は後回し。
- Day 1 未完のまま Day 2 → 朝イチで講師が該当者の [2]/[3] を一緒に片付ける。追いつかない場合は玄米茶の成果物を配って [4] から合流させる。

**15分ルール**: 15分自力で解決しなければ講師か隣の人に聞く（沼を防ぐ）。

## 4. 事前課題の提出・レビュー（R8-12）

「題材持ち寄り」は当日の最大の分散要因。**開催前に提出**させ、講師が1人5分で確認するだけで Day 1 の失敗率が大きく下がる。

- **提出物**: 玄米茶の必須機能5つ（LP / 価格 / 問い合わせ / マイページ / ライセンス配布）を**自分の題材に写像した表**（＝読み替え表）。締切は開催の数日前。
- **講師のチェック観点**:
  - サブスク課金（月額/年額）が自然に乗る題材か。
  - 5つの必須機能に無理なく写像できるか。
  - 3日で1周できる規模か（盛りすぎていないか）。
- 写像できない／複雑すぎる場合 → **「玄米茶に切り替え」を事前に案内**する。判断基準を事前に共有しておくと当日の停滞が減る。
- あわせて、[1] の玄米茶デモ画面を事前生成しておく（→ 「1. デモ構造」）。
- **開催直前チェック**: デモ URL（玄米茶サンプル）がブラウザで開けるか確認する（Worker が停止・上書きされていないか）。

## 5. [1] の生成待ちを活用する（SCREEN_SPEC 共有）

Claude Design の生成は数分かかる。この待ち時間を次のように使う。

- 参加者に **SCREEN_SPEC.md（自分のサービスの画面仕様）を書く/仕上げる**よう促す（`flow.html` [1] にも「生成待ちに書き始めてよい」と記載済み）。「画面構成とナビゲーション」の玄米茶例と同じ形式で自分の題材に読み替えて書く ＝「要件を先に `.md` 化してから AI に渡す」という本ワークショップ中核スキルの実践になる。
- 講師は**終わっていそうな人を数名指名**し、SCREEN_SPEC を画面共有してもらう。ねらい:
  - 待ち時間が埋まる。
  - 全員が別の SaaS を作る面白さが初日から見える（相互刺激）。
  - **題材が必須機能5つに写像できているか**をその場で確認できる（盛りすぎ・写像不能に早期に気づく＝ 「4. 事前レビュー」の当日版）。
- **オンライン開催なので「隣と見せ合い」はしない**。指名は進捗を見て**講師の裁量**で行い、参加者資料（`flow.html`）には固定ステップとして書かない。
- **休憩**: 固定枠は設けず、**生成待ちが長い場面（特に Day 2 [4]）で講師が小休止の号令**をかける（`flow.html` のタイムテーブル注記と対応）。
