3Days ワークショップフロー: AI とつくる「自分の SaaS の MVP」

完成形の動くデモ（玄米茶サンプル）: https://genmaicha-back-demo.y-arai-a8a.workers.dev/ — この3日間で、このような「LP（ランディングページ＝サービス紹介の1枚ページ）＋購入＋マイページ」の SaaS を自分の題材で作る。

ワークショップの目的・ゴール

前提知識（この3日間で前提とすること）

  • Claude Code を使ったことがある（起動・依頼・承認/拒否の基本操作が分かる）
  • Git を用いた開発経験がある（ブランチ・commit・push が一通りできる）
  • ソフトウェア開発の基本知識（基本情報技術者試験程度）

目的

  • Claude（Design / Code）を使った開発フローを3日間で一周する
    • 「AI にどう頼むと良いものが出てくるか」の肌感をつかむ
  • デザイン → フロント → バック → テスト → デプロイ → 機能拡張 という実務に近い開発サイクルを、AI を駆動しながら一通り体験する

題材: 各自が作りたい SaaS の MVP（事前に考えてきてもらう）

  • 題材は各自が作りたい SaaS を持ち寄る。「流れを体験するのが目的」なので題材は自由だが、サブスクリプション（月額/年額課金）を必須とする
  • ただし3日間で一周できるよう、MVP（Minimum Viable Product＝必要最小限の機能に絞った最初の製品版）に落とし込んでおく
  • 資料中の例「玄米茶」は説明用の共通サンプル: 「チャットの中に、ちいさな相棒を」という会話アシスタントを、ライセンスキーの月額/年額サブスクで売る小さな SaaS
    • 自分の題材では、このページ構成・機能を自分のサービスに読み替える
  • 注意喚起: 3日間で一周するため、Day 2 までは構成を盛りすぎない（拡張は Day 3 のオプションで）

必須機能（Day 2 までに作るものの最低ライン）

ライセンスキー（購入の証として発行する一意の文字列）の配布は、題材によらず全員共通の実装要件。「ライセンスキー」は玄米茶サンプルでの呼び方なので、自分の SaaS では呼び名と意味づけ（アクセス権の証・ダウンロードコード等）だけを自分のサービスに合わせる。

Claude Code のセキュリティ対策（最初に1回）

大前提（3日間ずっと効く）: 秘密情報（API キー・トークン）は Claude への指示にも会話にも貼らない。.dev.vars に入れて参照させる。AI が書いたコードも読まずに承認しない。

秘密を Claude に読ませない設定

よくある勘違い: 「.claudeignore を作る」→ Claude Code に .claudeignore は無い。正しくは .claude/settings.json の permissions.deny で「読み取り禁止」を宣言する。

プロジェクト直下に .claude/settings.json を作り、秘密ファイルへの Read を拒否しておく（.claude/ ディレクトリがなければ mkdir -p .claude で作成）:

{
  "permissions": {
    "deny": [
      "Read(./.dev.vars)",
      "Read(./**/.dev.vars)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./**/.env)",
      "Read(./**/.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
  • deny は最優先。万一「.dev.vars 読んで」と頼んでも Claude は読めない＝事故防止
  • 秘密ファイル（.dev.vars）が実際に登場するのは [4] バックから。設定だけ先に置いておくのがコツ

security-guidance プラグインを入れておく

security-guidance: Claude が生成したコードをバックグラウンドで自動セキュリティレビューしてくれるプラグイン。SQL インジェクション（等）・XSS（悪意あるスクリプトを画面に混入させる攻撃）・SSRF（サーバーに内部への不正リクエストをさせる攻撃）・IDOR（他人のデータを覗ける穴）・認可バイパス（ログインできた人が権限外のデータ・操作に届いてしまう穴）など 25 種類以上の脆弱性クラスを継続的に見てくれる。「AI が書いたコードを AI が点検する」を自動でやってくれるのがポイント。

メニューから入れる。まず次を実行:

/plugin

選択したら、有効化のため次を実行:

/reload-plugins

  • インストール後、リストに security-guidance が出るか確認する:

npm パッケージの取り込みを制限する

公開直後のパッケージは、乗っ取られたメンテナー権限で悪意あるコードが仕込まれていても発覚が追いついていないリスクがある。プロジェクト直下に .npmrc を作り、公開から7日未満のパッケージを取り込まない設定と、インストール時の既知脆弱性チェックを必須化する設定を入れておく。

min-release-age=7
audit=true
  • min-release-age=7: 公開7日未満のバージョンは npm install の対象外にする（typosquatting（名前の似た偽パッケージ）・乗っ取りアカウントからの悪性パッケージが広まりやすい初動期間を避ける）
  • ⚠️ min-release-age は npm 11.10.0 以降で有効。推奨の Node 24 LTS なら npm 11 系が同梱される（Node 20/22 は npm 10.x 同梱で無視されて効かない）。いずれにせよ必ず npm -v で確認し、11.10.0 未満なら npm install -g npm で更新する
  • audit=true: npm install 実行時に自動で npm audit 相当の脆弱性チェックが走る（デフォルトで有効だが、明示しておくと誤ってオフにされにくい）
  • AI に依存パッケージを追加してもらう場面（[4] バック実装など）でも、この設定は自動的に効く

[3] Day 1 フロントデプロイ（Cloudflare Workers）

ゴール: フロントが完成したらすぐに公開 URL を手に入れる（バックは Day 2 で繋ぐ）

Cloudflare Workers は静的ファイル（HTML・CSS・JS・画像など、ビルド済みでそのまま返す配信物）も配信できる。今日はフロントだけを Worker としてデプロイし、Day 2 で API を追加して同じ Worker に統合する。デプロイ先は最初から workers.dev の 1 URL に統一する。

流れ

手順1〜2（インストール〜 whoami）はブラウザ連携とアカウント確認があるため自分のターミナルで打つ。手順3〜4（ビルド〜デプロイ）は Claude Code に「ビルドしてデプロイして」と任せてよい（下のコマンドは自分で打つ場合の参考）。

  1. wrangler（Cloudflare の CLI ツール。デプロイやクラウド資源の作成をコマンドで行う）をグローバルインストールする: 
    npm install -g wrangler
  2. Cloudflare にログインする: 
    wrangler login     # ブラウザが開く → Cloudflare 上で「許可」する
wrangler whoami    # ログインできたか／想定アカウントかを確認
    • 会社用・個人用など複数アカウントを持っている人は whoami で取り違えていないか必ず確認（別アカウントに資源を作る事故を防ぐ）
  3. フロントのルートにいることを確認し、ビルドする: 
    npm run build
  4. Cloudflare Workers にデプロイ: 
    wrangler deploy

    初回は wrangler.jsonc の name が Worker 名として登録される。2回目以降は同じコマンドで上書きデプロイされる。また、アカウントで初めての Worker デプロイでは workers.dev のサブドメイン（URL の <account> 部分）の登録を求められることがある——画面の指示に従って任意の名前を登録すればよい。

  5. 発行された URL をブラウザで開いて LP が表示されることを確認 🎉（実際の URL は https://<name>.<アカウントのサブドメイン>.workers.dev の形。本資料では以降、短く https://<name>.workers.dev と表記する）

[4] Claude Code でバック作成

ゴール: フロントと API がつながり、ローカルで一通りの機能が動く

大事な前提：バックはモノレポ内の backend/ で作業する
[2] と同じくリポジトリのルートで Claude Code を起動し、backend/ に実装させる（フロントとバックを跨ぐため、ルート起動のまま進めるのが楽）。最終的に配信は Worker 一本にまとめる（手順5）。

流れ

  1. Hono + Cloudflare Workers のプロジェクトを Claude Code に作らせる（backend/）
    • 成果物: backend/ に Hono の雛形（src/index.ts・wrangler.jsonc・package.json）が揃った状態
    • npm run dev を実行し、表示された URL に GET / して返ってくれば OK（wrangler dev が使うポートは環境により変わるので、ターミナルに表示された URL を使うこと）。npm run dev などの script 定義は下の必要な npm scriptsにまとめてある（無ければ Claude Code に追加させる）
    • .gitignore も一緒に作らせる（node_modules / dist / .dev.vars などを除外）
    • プロンプト例: 
      backend/ に Hono + Cloudflare Workers の API プロジェクトを作って。
- src/index.ts・wrangler.jsonc・package.json を揃える
- まずは GET / が動作確認用のレスポンスを返す最小構成でよい
- .gitignore も作って（node_modules / dist / .dev.vars を除外）
  2. クラウド資源（DB・KV）を作って wrangler.jsonc に登録する
    • D1 … Cloudflare の SQLite データベース。問い合わせ・ライセンスなどを保存する
      KV … キーと値のペアを保存する軽量ストア。高速に読み書きできる。D1（SQLite）は書き込みが比較的重いため、「同じ IP から短時間に何回アクセスがあったか」のような一時的なカウンタには向かない。KV はそういった軽量な読み書きに適しているため、このアプリではレート制限のカウンタとして使う（なお KV は書き込みの反映に少し遅れがある「結果整合」のため、厳密なレート制限には本来不向き——Cloudflare 公式は Durable Objects を推奨。今回は簡易実装と割り切って使う）
    • Claude Code にまとめて任せる: 
      Cloudflare の D1 と KV を作成して、wrangler.jsonc に書き込んで。
なお、wrangler.jsonc の name はフロントの wrangler.jsonc と同じにして。

      name をフロントと同じにする理由: Day 2 でバックをデプロイすると同じ name の Worker を上書きする形になり、Day 1 で手に入れた URL（https://<name>.workers.dev）がそのまま引き継がれる。name が違うと別 URL になってしまう。

    なぜローカルで使わないのに先に作るのか？ — wrangler.jsonc に実在の database_id / KV id が書かれていないと、[6] のリモートデプロイが失敗するから。ローカル開発自体はクラウドの D1/KV に触れないので、ローカルとクラウドのデータは別物（詳細は手順4）。

  3. API をまとめて実装させる
    • 必要な機能を日本語でまとめて伝え、Claude にいっぺんに実装させてよい。API のパス・入出力の設計も基本 AI に任せるが、完了判定に使う POST /api/contact と GET /api/me の2本だけはこのパスに固定する
    • この題材で要る機能の例:
      • 問い合わせを受け付けて D1 に保存する（同一 IP から1分以内に3回以上来たら KV でカウントして 429（リクエスト過多）を返す）
      • メールアドレスを受け取ってライセンスキーを発行する（決済なし・Day 2 の簡易版。Day 3 [8] で Stripe 購入 → Webhook 発行に差し替え）

[5] 自動テストの実装

ゴール: 「正しいリクエストは通る・壊れたリクエストは弾く・ログインなしで保護 API に入れない」を npm test で自動確認できる状態にする。デプロイ前にここを緑にする

考え方（AI との壁打ち）

テスト観点を人間が決め打ちすると「表に無い＝テストしない」になりがち。何をテストするかも AI に出させ、人間はそれをレビューするのが抜けを防ぐコツ。

このワークショップでは3ステップの壁打ちでテストを作る:

  1. AI にテスト戦略を書かせる — コードを読んで「何を・なぜテストするか」を docs/TEST.md にまとめさせる
  2. 人間が TEST.md をレビューする — 観点の抜けや「これはテストしなくていい」という箇所をチャットで指摘する
  3. AI にテストを実装させる — レビュー済みの TEST.md を元に実装させ、npm test でグリーンを確認する

Resend・Stripe などの外部サービスはテスト中に呼ばない。実際の API を叩くとテストが遅くなり外部障害の影響も受けるため、fetch はモックに差し替える。

流れ

  1. 以下のプロンプトで TEST.md（戦略書）だけを作らせる: 
    このバックエンド（backend/）のコードを読んで、テスト戦略を docs/TEST.md にまとめてください。
テスト戦略には「何を・なぜテストするか・しないか」の判断も含めること。
テストファイルの実装はまだしないでください。
  2. 人間が TEST.md をレビューする — docs/TEST.md を自分で開き、最低限以下の4観点が含まれているかを確認する:
    • 外部サービスに依存しない純粋な関数のテスト
    • 正しい入力で API が成功レスポンスを返すテスト
    • 不正な入力で API が適切にエラーを返すテスト
    • 未認証でログインが必要な API にアクセスしたとき、拒否されるテスト

    抜けている観点があれば Claude Code に伝えて追記させる:

    TEST.md に「未認証でログイン必須 API にアクセスしたとき拒否される」観点が抜けているので追記して。

    時間に余裕があれば、4観点のチェックだけでなく全文に目を通し、「これはテストしなくていい」という箇所の指摘まで行うのが本来のレビュー。

  3. TEST.md が固まったらテストファイルを実装させる: 
    docs/TEST.md の方針に従ってテストを実装してください。

テストは @cloudflare/vitest-pool-workers を使い、設定は cloudflareTest プラグイン方式で書いて。
外部サービスへの fetch はすべてモックしてください。
    • @cloudflare/vitest-pool-workers＝テストを Workers と同じ実行環境で動かすための公式アダプタ。プロンプトのこの指定はそのままコピペでよい（設定ファイルの中身は AI が書く）
  4. npm test でグリーンを確認

ポイント

  • AI が書いたコードこそテストが要る（出力を鵜呑みにせず、テストで裏付ける）
  • TEST.md が先、実装が後（方針を固めてから書かせると精度が上がる）
  • Day 3 でオプション機能を追加するとき、このテストが安全網になる

送信ドメインはどうする？（ここがつまずきやすい）

独自ドメインから送るには DNS 検証（ドメインの所有者であることを DNS レコードの設置で証明する作業）が必要で、それは自分のドメインが決まる公開後の話。ワークショップでは次の方針を推奨：

  • 動作確認は Resend のテスト用アドレス onboarding@resend.dev を From に使う
    • 「自分宛に届く」を確認するには十分
  • 独自ドメインからの送信は任意（やりたい人だけ、デプロイ後に DNS を設定）

重要: この From はドメイン検証なしで送れるが、宛先は Resend に登録したアカウントのメールアドレス1つだけに限られる。それ以外のアドレスに送ると Resend がエラーを返す。

API キー作成時のスコープ設定について（本ワークショップでの方針）
Resend の API キー作成画面では「Sending domain」を絞り込めますが、本ワークショップでは "All domains" を選んで作成してください。独自ドメインを別途追加・DNS 検証しないと特定ドメインのキーが発行できず、手順が止まってしまいます。

[10] 運用保守★☆☆

お題: 公開後の運用を体験する（監視・ログ・障害特定・対策・コスト）
実装先: .claude/skills/（運用スキルを Claude Code に作らせる）
学び: 公開後の「運用」を Claude Code のスキルで仕組み化する

まず: Cloudflare API トークンを用意する

wrangler だけだと取得できる情報に制限がある。ログ集計やコスト試算には API トークンが要る。

  1. Cloudflare ダッシュボード右上のアイコン → Profile を選択
  2. 左メニューの API Tokens → Create Token をクリック
  3. 「Create Custom Token」の Get started をクリック
  4. Token name を入力し、Permissions を Account / Account Analytics / Read にして作成
  5. 取得したトークンを backend/.cf-ops.token に保存（例 CLOUDFLARE_API_TOKEN=... の1行）。.dev.vars とは別ファイルにする（理由は下記）。.gitignore に .cf-ops.token を追加しておく

このトークンは Worker が使うのではなく、運用スキルが shell からファイルを読んで使う（Cloudflare の集計 API（GraphQL）を叩くため）。ここで注意: .dev.vars は Day 1 の設定で Read を拒否しているので、スキルが cat 等で読もうとするとブロックされる。そこで運用トークンだけは deny 対象外の別ファイル（.cf-ops.token）に分離する。ここに入れるのは Analytics 読み取り専用の低権限トークンで、Stripe/Clerk 等の高価値シークレット（.dev.vars）とは分けて扱う。本番 secret に入れる必要はない（ローカルにあれば十分）。トークンが無くても wrangler tail のライブログは見られる。集計・コスト試算だけがトークン必須。

運用チェックを「スキル」にする

Claude Code のスキルとは、.claude/skills/<名前>/ ディレクトリに必要なファイルを置くことで「運用チェックして」のような自然言語で呼び出せる専用コマンドを登録できる機能。[4] で使った /code-review などの「組み込みスキル」は最初から入っているスキル、Day 1 で入れた security-guidance のような「プラグイン」はスキル等をまとめて配布するパッケージ、という関係。毎回同じ確認を Claude Code のスキル（.claude/skills/）にしておけば、「運用チェックして」の一言で再利用できる。作ってもらうスキルは5つ：

#スキルやること置き場所
1インフラ障害確認依存サービス（Cloudflare/Clerk/Stripe/Resend）の公開ステータスを確認.claude/skills/infra-status/
2ログ取得Cloudflare のログ／エラーを取得して問題がないか確認.claude/skills/cf-log-check/
3障害箇所の特定ログ・直近デプロイをソースと突き合わせて原因を絞る.claude/skills/fault-localize/
4対策・回避策ログ内容から恒久対策・暫定ワークアラウンドを出す.claude/skills/remediate/
5コストモニタリングrequests / CPU 時間からコストを試算.claude/skills/cost-monitor/

5つを一気通貫で回す統合スキル ops-check にまとめてもよい（.claude/skills/ops-check/）。いずれも Claude Code に作らせ、自分のサービス向けに調整する。

手順

  1. 上記の手順で API トークンを作成し、backend/.cf-ops.token に保存する（.dev.vars とは別ファイル。理由は上記）
  2. スキルを Claude Code に作ってもらう → 「運用チェックして」で実際に動くか確認 
    .claude/skills/ 以下に、本番運用チェック用のスキルを5つ作って。
1. infra-status: Cloudflare/Clerk/Stripe/Resend の公開ステータスページを確認して障害有無をまとめる
2. cf-log-check: wrangler tail で本番 Worker のログ・エラーを取得して問題がないか確認する
3. fault-localize: ログと直近のデプロイ差分をソースと突き合わせて障害箇所を絞り込む
4. remediate: 特定した原因への恒久対策と暫定ワークアラウンドを提案する
5. cost-monitor: Cloudflare の GraphQL Analytics API で requests / CPU 時間を集計してコストを試算する

    1. 点検対象を main ブランチに push（SonarCloud は main / PR を解析）
    2. https://sonarcloud.io で GitHub SSO ログイン
    3. 右上の ＋ → Analyze new project

⚠️ ハマりどころ

  • SonarCloud の指摘には誤検知（false positive）もある。鵜呑みにせず AI と判断する

  • 意図しないリポジトリまで勝手に解析されることがある。原因は2つの組み合わせ:
    • SonarCloud の GitHub App を All repositories（全リポジトリ）で許可している
    • SonarCloud の Organization で新規リポジトリの自動取り込み（Automatic Analysis / auto-provisioning）が有効

    • GitHub → Settings → Integrations → Applications → SonarCloud → Configure（https://github.com/settings/installations）で Only select repositories に変更して対象を絞る
    • 個別プロジェクトだけ止めたいなら、SonarCloud の対象プロジェクト → Administration → Analysis Method の「Automatic Analysis」を OFF、または Administration → Deletion で削除

[12] テスト設計（自分で決める）★★★

お題: 自分のアプリで「間違うと一番痛い」機能を1つ選び、その正しい振る舞いを自分で仕様化して、そこから要るテストを自分で決めて実装する

⚠️ 一番の罠: AI に「テスト書いて」と丸投げすると、AI は実装を読んで「今のコードがやっていること」をテストにする＝AI のバグごと正解として固定してしまう。だから機能テストは、まず意図した仕様（自分の言葉／SCREEN_SPEC.md）から導き、実装と突き合わせる。この順序が逆になった瞬間、テストは実装追認装置になる。

3Days ワークショップフロー — AI とつくる「自分の SaaS の MVP」

付録: Claude Code 認証切り替えガイド

対象: 普段は API キーで使っている人が、一時的に Claude Code サブスクに切り替え、用が済んだら API キーに戻すための手順。

これだけ覚える: どのキーを使うかは ~/.claude.json の記録で決まる。 rejected にあるキーは無視されてサブスクが使われ、approved にあるキーは使われる（サブスクより優先）。 切り替えとは、この記録を /config や起動時プロンプトで切り替えること。

認証の記録は ~/.claude.json にある

「どのキーを使う / 使わない」は ~/.claude.json の customApiKeyResponses に記録される。 approved（使うキー）と rejected（使わないキー）の2つの配列があり、キーは末尾20文字だけが記録される（フルキーは保存されない）。

キーの3つの状態

状態意味起動時の挙動
approvedこのキーを使う、と承認済みこの API キーで動く（サブスクより優先）
rejectedこのキーは使わない、と拒否済みキーを無視してサブスクで動く
newまだどちらも選んでいない「このキーを使う？」と確認が出る

① API → サブスク: 一時的にサブスクへ切り替える

普段の API キー運用から、一時的にサブスクを使い始めるとき。

  1. Claude Code のセッション内でログアウトする（approved が空になり、古いサブスクのログインも消える）。 
    /logout
  2. Claude Code を起動しなおす。env に API キーが残っているので「このAPIキーを使うか？」と聞かれる。ここで 「2. No (recommended)」を選ぶ。 
    claude
  3. No を選ぶと、ログイン方法の選択画面が自動で表示される（/login を打つ必要はない）。「1. Claude account with subscription」を選び、ブラウザで claude.ai（Pro / Max 等）にログインする。
  4. 認証方式を確認する。claude.ai になっていれば成功。~/.claude.json は approved が空・rejected に当該キーが入った状態になる。 
    /status

環境変数の API キーは 消さなくてよい（/logout が承認を外すのでサブスクが使われる）。 ただし起動のたびに確認プロンプトが出るのが煩わしければ、そのシェルで unset ANTHROPIC_API_KEY しておくとプロンプトが出なくなる。

② サブスク → API: 普段の運用に戻す

サブスクの用が済んで、普段の API キーに戻すとき。ゴールはキーを rejected から外すこと。/config の ON で rejected から外れ、approved として記録される。

  1. env に API キーがあることを確認する（無ければ設定）。普段 ~/.zshrc 等に書いてあるならそのままでよい。 
    export ANTHROPIC_API_KEY="sk-ant-xxxxxxxx"
  2. Claude Code を起動する。 
    claude
  3. API キーを有効化する。① の直後はキーが rejected に入っているため起動時プロンプトは出ない。/config を開き「Use custom API key」を ON にする（キーが未決定=new の場合は起動時に確認が出るので、そのときは「1. Yes」でもよい）。 
    /config
  4. 認証方式を確認する。API キー利用になっていれば成功。~/.claude.json は approved に当該キーが入り rejected が空の状態に戻る（冒頭の ~/.claude.json の例と同じ状態）。 
    /status

一度「使わない」を選んでいると、キーは rejected に入っている。 /config の ON で自動的に approved へ移るので問題ない。 手動でやるなら ~/.claude.json の customApiKeyResponses.rejected から 該当キーの末尾20文字のエントリを削除する（削除すると new に戻り、再度確認プロンプトが出る）。

確認とトラブルシュート

確認に使うコマンド

現在の認証方式を見る（Claude Code セッション内）:

/status

approved / rejected の登録件数を確認する（キーの値は表示されない）:

jq '.customApiKeyResponses | {approved:(.approved|length), rejected:(.rejected|length)}' ~/.claude.json

うまく切り替わらないとき

症状原因対処
サブスクにしたのに API のままキーが approved に残っている/logout、または /config のトグルを OFF
API にしたいのに使われないキーが approved に無い（new/rejected）/config を ON、または起動時プロンプトで「使う」
ログインしても接続が切れるANTHROPIC_AUTH_TOKEN / CLAUDE_CODE_OAUTH_TOKEN が env に残存（承認ゲート無しで効く）該当変数を unset して再起動
/logout しても API が消えない/login で入力・保存したキー（keychain）は logout で消えない/status でソースを確認し、/login でサブスクに入れ直す

秘密情報の扱い: API キーの値をチャットやコードに貼らない。git 管理下のファイルにも書かない。恒久設定は各自の ~/.zshrc 等にとどめる。

仕組み（詳細・読まなくても切り替えはできます）

認証の優先順位

複数の認証情報があると、上ほど優先される。

順位認証方式
1クラウドプロバイダの資格情報（Bedrock / Vertex 等）
2ANTHROPIC_AUTH_TOKEN（Bearer トークン）
3ANTHROPIC_API_KEY — ただし approved のときだけ採用
4apiKeyHelper スクリプト
5CLAUDE_CODE_OAUTH_TOKEN
6サブスクの OAuth（/login で保存）

ANTHROPIC_API_KEY は承認ゲートがあり、未承認なら無視されてサブスクにフォールバックする。 一方 ANTHROPIC_AUTH_TOKEN / CLAUDE_CODE_OAUTH_TOKEN はゲートが無く、セットされていれば効く。

/logout が実際にすること

  • customApiKeyResponses.approved を空にする（承認済みキーが外れる）
  • サブスクの OAuth 資格情報を消す
  • オンボーディング状態をリセット → 次回起動で確認プロンプトが再表示される
  • rejected には何も追加しない。/login で保存した keychain 上のキーも消さない

2つの「キーの保管場所」

  • 環境変数 ANTHROPIC_API_KEY … 承認（approved/rejected）で制御。本ガイドの前提。
  • 保存キー primaryApiKey（macOS Keychain）… /login でキーを直接入力した場合の保管先。/logout では消えない。env 運用なら基本使われない。

補足: --bare / CLAUDE_CODE_SIMPLE などの簡易モードや非対話実行（claude -p）では、 確認プロンプトを出さずに env のキーを直接使う経路がある（＝承認ゲートを通らない）。通常の対話起動は承認ゲート経由。
