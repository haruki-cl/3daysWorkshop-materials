3Days ワークショップフロー: AI とつくる「自分の SaaS の MVP」

ワークショップの目的・ゴール

前提知識（この3日間で前提とすること）

  • Claude Code を使ったことがある（起動・依頼・承認/拒否の基本操作が分かる）
  • Git を用いた開発経験がある（ブランチ・commit・push が一通りできる）
  • ソフトウェア開発の基本知識（基本情報技術者試験程度）

目的

  • Claude（Design / Code）を使った開発フローを3日間で一周する
    • 「AI にどう頼むと良いものが出てくるか」の肌感をつかむ
  • デザイン → フロント → バック → テスト → デプロイ → 機能拡張 という実務に近い開発サイクルを、AI を駆動しながら一通り体験する

題材: 各自が作りたい SaaS の MVP（事前に考えてきてもらう）

  • 題材は各自が作りたい SaaS を持ち寄る。「流れを体験するのが目的」なので題材は自由
  • ただし3日間で一周できるよう、下記の必須機能に相当するものを備えた MVP に落とし込んでおく（事前周知で「どんな SaaS を作るか・必須機能をどう満たすか」を軽く考えてきてもらう）
  • 資料中の例「玄米茶」は説明用の共通サンプル: ライセンスキー販売型のサブスク SaaS
    • ページ構成: LP（/）・問い合わせ（/contact）・マイページ（/mypage）・購入完了（/success）
    • 自分の題材では、このページ構成・機能を自分のサービスに読み替える
  • 注意喚起: 3日間で一周するため、Day 2 までは構成を盛りすぎない（拡張は Day 3 のオプションで）

Git ブランチ（dev → main）

前提: GitHub でリポジトリを新規作成し、ローカルに clone してから以下を実施する。フロント・バックは同じリポジトリ（モノレポ）で管理する。

SonarCloud はモノレポのリポジトリを丸ごと解析できる。

  • 最初に dev ブランチを切り、作業はすべて dev で進める（空リポジトリの場合は README 等を main に最初の commit で push してから dev を切る）
  • 区切りのいいところでこまめに commit / push（環境が壊れても作業を失わないための保険）
  • 各日のクロージングで「dev に push 済みか」を必ず確認（翌日のキャッチアップが楽になる）
  • 完成したら dev → main にマージ（main ＝ 公開・解析対象の安定版）
git switch -c dev      # 最初に dev を切る
git push -u origin dev  # 初回 push は -u で upstream を設定
# … dev で実装・こまめに commit / push …
git switch main
git merge dev          # 完成したら main へ
git push origin main   # main もリモートへ push
  • SonarQube は main（または PR）を解析する想定（[11] セキュリティで使う）
  • 秘密ファイル（.dev.vars 等）は .gitignore で必ず除外してコミットしない

Claude Code のセキュリティ対策（最初に1回）

大前提（3日間ずっと効く）: 秘密情報（API キー・トークン）は Claude への指示にも会話にも貼らない。.dev.vars に入れて参照させる。AI が書いたコードも読まずに承認しない。

秘密を Claude に読ませない設定

よくある勘違い: 「.claudeignore を作る」→ Claude Code に .claudeignore は無い。正しくは .claude/settings.json の permissions.deny で「読み取り禁止」を宣言する。

プロジェクト直下に .claude/settings.json を作り、秘密ファイルへの Read を拒否しておく（.claude/ ディレクトリがなければ mkdir -p .claude で作成）:

{
  "respectGitignore": true,
  "permissions": {
    "deny": [
      "Read(./.dev.vars)",
      "Read(./**/.dev.vars)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)"
    ]
  }
}
  • deny は最優先。万一「.dev.vars 読んで」と頼んでも Claude は読めない＝事故防止
  • 秘密ファイル（.dev.vars）が実際に登場するのは [4] バックから。設定だけ先に置いておくのがコツ

security-guidance プラグインを入れておく

security-guidance: Claude が生成したコードをバックグラウンドで自動セキュリティレビューしてくれるプラグイン。インジェクション・XSS・SSRF・IDOR・認可バイパスなど 25 種類以上の脆弱性クラスを継続的に見てくれる。「AI が書いたコードを AI が点検する」を自動でやってくれるのがポイント。

/plugin install security-guidance@claude-plugins-official

※ コマンドが見つからない場合はスキップ可。Day 1〜2 の本編には影響しません。

  • インストール後、/plugin list で security-guidance がリストに出ることを確認
  • 前提: python3 が PATH に通っていること
  • うるさく感じたら SECURITY_GUIDANCE_DISABLE=1 で全体オフできる（キルスイッチ）

[1] Claude Design でサイトデザイン作成

ゴール: LP および主要画面（マイページ・問い合わせ・サクセス）の雰囲気・カラー・セクション構成の方向性を決める（ピクセルパーフェクトなカンプは作らない）

アクセス: Claude Design は claude.ai にブラウザでログインして使う（Claude Code と同じアカウント）。左サイドバーの「Explore Claude」→「Claude Design」を選ぶか、https://claude.ai/design に直接アクセス。

なぜ Design.md を作らないのか
UX デザインの一般的なフローは「Design.md で構造・要件を言語化 → Claude Design でビジュアライズ」だが、今回は省略している。理由は、LP・問い合わせ・マイページ・購入完了というごく一般的な SaaS の画面構成であり、Claude Design 自体がそのパターンを熟知しているため、事前に言語化しなくても十分なアウトプットが出るから。Design.md が効くのは独自のユーザーフロー・複雑な要件・チームでの認識合わせが必要な場合。今回はいずれも該当しないので、サービスの内容を直接伝えるだけで十分。

時間短縮のコツ: 最初は他ページを盛らず LP だけで3案 → 1つ選ぶ → そのあとに他ページ（マイページ・問い合わせ・サクセス）を追加生成する

  1. 出てきた3案を見比べて1つ選ぶ
    • 最初の1案でなんとなく進めず、3案を見比べてから決める（選択肢を見てから選ぶのが AI 活用のコツ）
  2. トーン・配色・アクセントは画面上部タブの「Tweaks」から後で調整できる（プロンプトに盛り込まなくても、触って変えられる）
  3. 生成されたデザインを見直し、不要なセクションや機能はチャットで削除・修正を依頼する
    • Claude Design はプロンプトの意図を超えて機能やセクションを追加することがある。「このセクションは不要です」「このボタンを消してください」とチャット欄で伝えれば修正してもらえる

    • 操作: 画面右上の「Share」ボタン → 「Project archive (.zip)」を選んでダウンロードする。
    • → これを [2] でそのまま Claude Code に渡す。色やフォントを別途テキストで言語化する必要はない（デザインそのものを渡す方が早くて正確）

[4] Claude Code でバック作成

ゴール: フロントと API がつながり、ローカルで一通りの機能が動く

大事な前提：バックはモノレポ内の backend/ で作業する
[2] で作成したリポジトリの backend/ ディレクトリで Claude Code を起動して作業する。最終的に配信は Worker 一本にまとめる（手順6）。

流れ

  1. Hono + Cloudflare Workers のプロジェクトを Claude Code に作らせる（backend/）
    • 成果物: backend/ に Hono の雛形（src/index.ts・wrangler.toml・package.json）が揃った状態
    • npm run dev で localhost:8787 が起動し、GET / が返ってくれば OK
    • .gitignore も一緒に作らせる（node_modules / dist / .dev.vars などを除外）
  2. クラウド資源（DB・KV）を作って wrangler.toml に登録する
    • D1 … Cloudflare の SQLite データベース。問い合わせ・ライセンスなどを保存する
      KV … Redis に近い NoSQL ストア。キーと値のペアで高速に読み書きできる（例: "ip:1.2.3.4" → 5回）。このアプリではレート制限のカウンタに使う
    • Claude Code にまとめて任せる: 
      Cloudflare の D1 と KV を作成して、wrangler.toml に書き込んで。
この時、wrangler.toml の name はフロントの wrangler.toml と同じにして。

      name をフロントと同じにする理由: Day 2 でバックをデプロイすると同じ name の Worker を上書きする形になり、Day 1 で手に入れた URL（https://<name>.workers.dev）がそのまま引き継がれる。name が違うと別 URL になってしまう。

    なぜローカルで使わないのに先に作るのか？ — wrangler.toml に実在の database_id / KV id が書かれていないと、[6] のリモートデプロイが失敗するから。ローカル開発自体はクラウドの D1/KV に触れないので、ローカルとクラウドのデータは別物（詳細は手順4）。

  3. API をまとめて実装させる
    • 必要な機能を日本語でまとめて伝え、Claude にいっぺんに実装させてよい。API のパス・入出力の設計も AI に任せる
    • この題材で要る機能の例:
      • 問い合わせを受け付けて保存する（同一 IP からの連投はレート制限で弾く）
      • ログイン中ユーザーの情報を返す（Day 2 はモック認証でよい。リクエストヘッダのメールを信頼して返すだけ。本物の Clerk は Day 3 [9]）
      • ライセンスキーを検証する
    • 保存するデータの中身（例: 問い合わせ＝名前／メール／本文）だけ先に決めておくと AI が迷わない。テーブル定義は AI に書かせ、migrations/ 以下に置くよう指示する
    frontend/ のコードを見て、それに対応するバックエンドを backend/ に実装して。
D1 と KV を使うこと。マイグレーションファイルは migrations/ 以下に作って。
  4. ローカル用の DB を用意する（マイグレーション）

    ローカルの D1（--local）とクラウドの D1（--remote）はデータが別物。開発中は手元だけを使い、本番への反映は [6] デプロイ時にまとめて行う。

    タイミングコマンド対象
    API 実装中（今ここ）npm run db:migrate:local手元の D1（ローカル）
    デプロイ直前（[6]）npm run db:migrate:remoteCloudflare の D1（本番）
     
    npm run db:migrate:local    # 手元(ローカル)の D1 にテーブルを作る
  5. フロントのモック表示を実 API 呼び出しに差し替える
    • 前提: backend で npm run build:front && npm run dev（:8787）を起動した状態で作業する
    • Claude Code への指示例: 「モックを実 API 呼び出しに変えて。バックは localhost:8787」
    • 差し替え後は http://localhost:8787 でブラウザ確認する
  6. 配信を一本化する（本番の形にする）
    • フロントを dist/ にビルドし、Worker から配信する（未知パスは SPA フォールバックで index.html）
    • フロントと API が同一オリジンになるため、本番では CORS 設定が不要になる
    • モノレポ内の frontend/ をビルドし、配信だけ1つの Worker にまとまるイメージ
    • Claude Code に「wrangler.toml に [assets] を追加してフロントの dist を配信させて」と依頼する

[5] 自動テストの実装

ゴール: 「正しいリクエストは通る・壊れたリクエストは弾く・ログインなしで保護 API に入れない」を npm test で自動確認できる状態にする。デプロイ前にここを緑にする

考え方（テスト戦略）

テストは3層。上から順に費用対効果が高い:

  1. バックエンド統合テスト — Hono app にリクエストを投げて検証（Vitest + @cloudflare/vitest-pool-workers、D1/KV は実バインディング）
  2. 純粋関数の単体テスト — キー生成・マスク処理など
  3. フロントのコンポーネントテスト — 状態分岐が多いものだけ（全網羅しない）

外部サービスへの fetch はモックする（テストを外部に依存させず、速く・安定して回すため）。

Claude Code へ渡すプロンプト例

「テスト観点の固定表」はあえて載せない。こちらが観点を決め打ちすると「表に無い＝テストしない」になりがち。どこを・どうテストするかも含めて AI に出させ、人はそれをレビューする方が、抜けが出にくい。

以下のプロンプトをそのまま Claude Code に渡して始めると効率的:

このバックエンド（backend）のテストを書いてください。

まず tests/ ディレクトリとテスト方針を docs/TEST.md にまとめてから実装してください。
カバーしてほしい範囲:
1. 純粋関数のテスト（キー生成・バリデーション等）— テスト環境の動作確認も兼ねる

⚠️ ハマりどころ: テスト環境の設定

Hono のバックは Node.js ではなく Cloudflare Workers という独自の実行環境で動く。そのため通常の Vitest（Node.js 上で動く）では D1・KV などの Workers 固有の機能が再現できず、テストが動かない。

@cloudflare/vitest-pool-workers を使うと、Workers のローカルシミュレーター上でテストが走り、D1・KV も含めた本番に近い状態で確認できる。設定の骨格は以下を Claude Code に渡すだけでよい:

import path from "node:path"
import { defineWorkersConfig, readD1Migrations } from "@cloudflare/vitest-pool-workers/config"

export default defineWorkersConfig(async () => {
  const migrations = await readD1Migrations(path.join(__dirname, "migrations"))
  return {
    test: {
      poolOptions: {
        workers: {
          wrangler: { configPath: "./wrangler.toml" },
          miniflare: {
            bindings: { TEST_MIGRATIONS: migrations, ORIGIN: "http://localhost:8787" },
          },
        },
      },
    },
  }
})

外部サービス（Resend・Stripe など）のキーはダミー文字列を渡す。テスト中に実際の送信・課金が走らないようにするため。test/setup.ts の作成は Claude Code に「vitest-pool-workers の test/setup.ts も作って」と依頼する。

    • 未認証でログインが必要な API にアクセスしたとき、アクセスが拒否されることを確認するテスト（例: ログインせずにマイページ API を叩いたら弾かれる）

送信ドメインはどうする？（ここがつまずきやすい）

独自ドメインから送るには DNS 検証が必要で、それは自分のドメインが決まる公開後の話。ワークショップでは次の方針を推奨：

  • 動作確認は Resend のテスト用アドレス onboarding@resend.dev を From に使う

  • 独自ドメインからの送信は任意（やりたい人だけ、デプロイ後に DNS を設定）

API キー作成時のスコープ設定について（本ワークショップでの方針）
Resend の API キー作成画面では「Sending domain」を絞り込めますが、本ワークショップでは "All domains" を選んで作成してください。独自ドメインを別途追加・DNS 検証しないと特定ドメインのキーが発行できず、手順が止まってしまいます。

  • From / To をどうするかも含めて AI に相談し、まずローカルで往復させる

    • Webhook で発行したライセンスは購入者の emailに紐づく。マイページはログイン中ユーザーの emailでライセンスを引く

  • 署名検証は「生のリクエストボディ」で行う（フレームワークが body をパースする前に処理する）

まず: Cloudflare API トークンを用意する

[11] セキュリティ★★★

お題: 自分のアプリをセキュリティ観点で点検し、最低1つ直す
ツール: Claude Code（/security-review・常時の security-guidance）＋ SonarCloud
参考: 本リポジトリ TEST.md の A 系ケース（認証境界の回帰テスト一覧）
学び: 「AI が書いたコード」を自分で点検できるようになる

見る観点

  • 認証境界: 未認証／偽造 JWT で 401 になるか
  • IDOR（Insecure Direct Object Reference：権限のない他人のデータを横断参照できてしまう脆弱性）: 他人の email を body に入れても無視されるか（他人のデータを引けないか）
  • 機密のマスク: API レスポンスにフルのライセンスキーが含まれていないか

2通りの点検を併用する

  • Claude Code
    • /security-review で差分・コードを点検させる
    • 事前に security-guidance を入れていれば、編集・コミットのたびに自動で点検されている（Claude Code セキュリティ対策参照）
  • SonarCloud（静的解析）

    SonarCloud とは GitHub と連携する静的解析 SaaS。コードを push するたびにセキュリティ脆弱性・バグ・コード品質の問題を自動検出し、Claude Code の差分レビューでは気づきにくいパターンを補完する。

    ⚠️ GitHub App のアクセス範囲について: SonarCloud の GitHub App 連携時は、必ず 「Only select repositories」 を選び、点検したいリポジトリだけを許可してください。「All repositories」を選ぶとプライベートリポジトリのコードが外部に送信される恐れがあります（詳細は下記ハマりどころ参照）。

    1. 点検対象を main ブランチに push（SonarCloud は main / PR を解析）
    2. https://sonarcloud.io で GitHub SSO ログイン
    3. 右上の ＋ → Analyze new project
    4. 対象の GitHub プロジェクトを指定（初回は SonarCloud にリポジトリへのアクセス許可を求められる）。
      • ここで GitHub App（SonarQubeCloud）のアクセス範囲を聞かれたら、All repositories ではなく Only select repositories を選び、点検したいリポジトリだけ許可する（下のハマりどころ参照）
    5. （上記で Only select repositories に設定済みであることを確認してから）解析方式は 「Automatic Analysis」 を選ぶと設定が楽（以後 push のたびに自動解析）
    6. 解析完了を待ち、出た指摘を AI と一緒にレビュー（意味・直し方を AI に説明させる）

⚠️ ハマりどころ

  • SonarCloud の指摘には誤検知（false positive）もある。鵜呑みにせず AI と判断する
  • 解析単位を分けたいので、専用リポジトリを対象にするのが推奨（Git ブランチ参照）
  • 意図しないリポジトリまで勝手に解析されることがある。原因は2つの組み合わせ:
    1. SonarQubeCloud の GitHub App を All repositories（全リポジトリ）で許可している
    2. SonarCloud の Organization で新規リポジトリの自動取り込み（Automatic Analysis / auto-provisioning）が有効
    → App が見える全リポジトリに push のたび SonarCloud が自動でプロジェクトを生成して解析する。プライベートリポジトリが含まれる場合はコードが外部に送られるため注意。誤って All repositories を選んでしまった場合の対処:
    • GitHub → Settings → Integrations → Applications → SonarQubeCloud → Configure（https://github.com/settings/installations）で Only select repositories に変更して対象を絞る
    • 個別プロジェクトだけ止めたいなら、SonarCloud の対象プロジェクト → Administration → Analysis Method の「Automatic Analysis」を OFF、または Administration → Deletion で削除
