## 資料の修正・改善案

### 1. Claude Design のモデル設定確認手順
**問題**: 資料に「左下で Claude Opus 4.8 / Effort: High を確認」とあるが、Claude Design のバージョン更新により UI が変わり、この指示が現状と合わない。  
**対応**: Claude Design の最新 UI に合わせて、モデル・プラン設定の確認手順を更新する必要がある。

### 2. Claude Design で参加者の自由度について明記がない
**問題**: Claude Design で質問（トーン・配色・料金プラン内容など）が出た際、参加者が「どこまで自由に決めていいか」が資料に明記されていない。  
**対応**: 「参加者が自由に決める部分」と「ワークショップ流の推奨値」の区別を明示する文言を追加する。

### 3. 料金プランの構成について指示がない
**問題**: Claude Design の質問で「料金プランの中身は？」と聞かれる。  
**対応**: 「最低限『月額と年額の2プラン』を用意すること」と明示すること。

### 4. Claude Design のデザインシステム初期化について未記載
**問題**: 資料に記載なし。実際には参加者の環境によって従来のデザインシステムが自動適用されてしまう可能性がある。  
**対応**: 「最初にデザインシステムをリセット／切っておくこと」を参加者指示として追記すること。

### 5. LP の構成要素（ヒーロー等）の用語定義がない
**問題**: 資料に「ヒーロー・特徴・料金・FAQ」と書かれているが、「ヒーロー」が何かの説明がない。  
**対応**: 用語（ヒーロー＝ページ最上部のメインビジュアル＋テキスト）を資料内で定義すること。

### 6. 「LP 一枚」という選択肢について明記がない
**問題**: Claude Design の質問で複数選択肢が出ることが明示されていない。  
**対応**: 「『LP 一枚』を選ぶこと」と明示すること。

### 7. Claude Design のデザイン案生成に時間がかかる
**問題**: 資料に「生成には数分かかる」とあるが、実際には 3 案の生成に 10 分以上かかることもある。  
**対応案**:
- デザイン生成中の待ち時間を活用して、次のタスクを前倒しで進める
- Day 1 のタイムテーブルにデザイン生成時間を明示
- 待ち時間中に参加者に parallel タスクを実施

### 8. 参加者のサービス仕様がバラバラなため、ページ構成・コンテンツが異なる
**問題**: 玄米茶の例は具体的だが、参加者が別の SaaS を作る場合の読み替え方法が不明確。  
**対応**: 参加者が題材を決めたら、Claude Code にプロンプトを渡す前に「自分のサービス向けに画面仕様を定義させるプリステップ」を追加する。

### 9. [2] フロント作成プロンプトが長大で、玄米茶固有の指示が混在している
**問題**: プロンプトにサービス固有の内容が埋め込まれており、他の題材の参加者が使いにくい。  
**対応**: プロンプトを「共通部分」と「サービス固有部分」に分離し、後者は SCREEN_SPEC.md を参照する形にする。

### 10. 参加者に Day 1 前に「画面仕様定義ドキュメント」を作成させるプリステップを導入
**背景**: フロント・バック実装がサービス固有の仕様に依存しているため、参加者が事前に定義すべき。  
**提案**: Day 1 開始前に `SCREEN_SPEC.md` を参加者に作成させ、以下をテンプレートとして定義させる：
- LP（/）: ヒーロー・特徴・料金・FAQ・CTA
- マイページ（/mypage）: 表示する情報ブロック
- 問い合わせ（/contact）: フォーム項目
- サクセス（/success）: メッセージ

**効果**: Claude Code への指示が明確になり、修正回数が減る。

### 11. フロント実装中の待ち時間を活用したタスク並列化
**背景**: フロント実装には 5 分以上かかる。この待ち時間を活用したい。  
**提案**: フロント実装中に `.claude/settings.json` を `frontend/` 直下に作成する。

**効果**: セキュリティ設定を早期に組み込める。

### 12. security-guidance プラグインのインストール手順が資料で不明確
**問題**: 2 つのコマンドが同じコードブロックに記載されており、参加者がまとめてコピーするとエラーが起きる可能性がある。  
**対応**: コマンドを別々のコードブロックに分離するか、「1 行ずつ実行してください」と明記すること。

### 13. npm run dev のポート指定について
**気づき**: `npm run dev` はポートが既に使用中の場合、自動で別ポートに割り当てる。「localhost:5173 を開く」という固定指示は不要。  
**対応**: 「`npm run dev` の出力に表示されたローカル URL を開く」という汎用的な記述に変更すべき。

### 14. Day 1 タイムテーブルの見直し
**問題**: Claude Design に 45 分が割かれているが、実際は 30 分程度で十分。フロント作成に時間が必要。  
**提案**: タイムテーブルを以下に調整：
- [1] Claude Design: 20〜30 分
- [2] フロント作成: 45〜50 分
- [3] フロントデプロイ: 15 分

**効果**: フロント実装に余裕ができる。

### 15. Day 2 バック実装時のポート指定について
**問題**: 資料で「:8787」という明示的なポート指定が書かれているが、`wrangler` が自動でバインドするため不要。  
**対応**: ポート指定を削除し、「`npm run dev` で表示されたローカル URL を開く」という汎用的な記述に変更すべき。Day 1 のポート指定削除と統一。

### 16. ローカル開発環境でのコマンド実行について
**気づき**: `npm run build:front && npm run dev` などのコマンドも Claude Code に任せた方がシンプル。  
**対応**: 参加者がコマンドを手打ちするのではなく、Claude Code に「ローカルで起動して動作確認して」と指示する形にすべき。

### 17. フロント実装時に「メールアドレスベースの簡易認証フロー」を明示すべき
**問題**: Day 1 [2] フロント作成時、資料のプロンプトに「メールアドレス入力での本人確認」が明確に指示されていない。  
**現状**: 
- マイページ表示: メールアドレス入力で本人確認（パスワード認証なし）
- 購入時: メールアドレス入力でライセンスキー発行

**対応**: Day 1 [2] のフロント実装プロンプトに以下を明示：
- 「マイページ（/mypage）へアクセス時、メールアドレス入力フォームで本人確認してから表示する」
- 「購入フロー（料金セクション → /success）でもメールアドレス入力を挟む」
- 「この段階ではパスワード認証は不要。メールアドレスのみで OK」

**効果**: フロント実装時に認証フロー（シンプルな形）が確実に実装される。

### 18. テスト戦略（TEST.md）のレビューを AI に任せる
**背景**: Day 2 [5] 自動テスト作成では、Claude Code が生成した TEST.md を人間が手動で確認し、「4 点の観点が含まれているか」をチェックしている。ワークショップの 2 時間という限られた時間では、この工程を AI に自動化する方が効率的。

### 19. Claude Code への git コミット指示を汎用化すべき
**問題**: 資料では各セクション完了時に「[セクション名]の作業をコミットして dev ブランチに push して」という具体的な指示をしているが、これだと指定したセクションの変更だけが push されるリスクがある。

**例**: 「自動テスト実装の作業をコミット」と指示すると、テストファイルだけが push され、それ以外の変更（バック実装、設定ファイルなど）が漏れる可能性がある。

**対応**: 各セクション完了後のコミット指示を以下のように統一する：
```
dev ブランチの全ての変更をコミットして、リモートに push してください。
```

あるいは：
```
現在の作業をコミットして dev ブランチに push してください。
```

**効果**:
- 変更の漏れを防ぐ
- Claude Code が「全ての変更」をまとめてコミット・push する
- git 履歴が明確になる

**実装タイミング**: flow.md 全体のコミット指示を汎用化する。

**実装タイミング**: flow.md の [6] デプロイセクションで、コマンド実行を Claude Code へのプロンプトに置き換える。
